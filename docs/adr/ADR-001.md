# ADR-001: Estratégia de Ingestão de Dados do ERP

**Status:** Aceito | **Data:** 2026-03-13 | **Autores:** Equipe de Arquitetura — Turma DAS II / UNIVILLE

---

## Contexto

A VendaMais Distribuidora Ltda. opera um ERP proprietário on-premises desde 2019 que centraliza transações de Vendas, Estoque, Financeiro e Logística (~3.500 pedidos/mês). O sistema não possui camada analítica nativa e os relatórios são gerados manualmente em Excel, causando defasagem de até 30 dias na tomada de decisão.

O objetivo do projeto é automatizar a extração diária de dados desse ERP para um pipeline na nuvem Azure, com defasagem máxima de 24 horas. Para isso, é necessário decidir qual mecanismo computacional será responsável por conectar ao ERP, extrair os dados e depositá-los no armazenamento cloud.

**Restrições e fatores relevantes:**

- A extração é diária e agendada (não precisa ser em tempo real)
- O volume é relativamente baixo: ~3.500 pedidos/mês, dados de 18 representantes, 1.200 clientes, com estimativa de 50 a 200 MB por extração diária
- A equipe é reduzida e não deve operar infraestrutura de servidores
- O projeto está em contexto acadêmico com orçamento restrito, então o custo deve ser mínimo e proporcional ao uso
- A solução deve ser totalmente na nuvem Azure (restrição do SOW-2026-001)
- A extração deve ser resiliente: falha em um dia não deve comprometer o histórico
- Não é permitido modificar o ERP ou sua base de dados

---

## Decisão

Adotar Azure Functions com Timer Trigger (modelo serverless, Consumption Plan) como mecanismo de ingestão de dados do ERP.

A função será desenvolvida em Python 3.11 e executada diariamente por agendamento cron (`0 2 * * *`, às 02h da manhã). Ela irá se conectar ao ERP via API REST ou ODBC (somente leitura), extrair os registros das últimas 24 horas de cada módulo (Vendas, Estoque, Financeiro e Logística) e salvar arquivos JSON/CSV na camada raw do Azure Blob Storage. Logs de execução e alertas de falha serão monitorados via Application Insights.

---

## Consequências

(+) **Custo praticamente zero em repouso:** No modelo Consumption Plan, a função só gera custo quando executa. Como a execução é diária e rápida, o custo tende a ficar dentro do free tier da Azure.

(+) **Sem necessidade de gerenciar infraestrutura:** Não há servidor, VM ou container para manter. A equipe pode focar apenas no desenvolvimento da lógica de extração.

(+) **Escalabilidade automática:** Caso o volume de dados aumente, a própria Azure escala a execução da função sem necessidade de intervenção.

(+) **Alinhamento com a arquitetura definida:** O SOW-2026-001 já prevê o uso de Azure Functions com Python na ingestão, o que facilita a integração com os demais componentes.

(+) **Desenvolvimento simples e rápido:** Funções com Timer Trigger são fáceis de desenvolver, testar localmente e implantar via CI/CD.

(-) **Cold start:** Pode haver um pequeno atraso na primeira execução após inatividade. Para esse cenário (execução diária), isso não impacta.

(-) **Limite de execução:** O Consumption Plan possui limite de tempo. Se o volume crescer muito, pode ser necessário migrar de plano ou dividir a carga em múltiplas funções.

(-) **Dependência do ERP:** A extração depende da disponibilidade do ERP no horário definido. Em caso de falha, será necessário implementar lógica de retry.

---

## Alternativas Rejeitadas

- **Azure Virtual Machine (VM) com script agendado:** Exige manter uma VM ativa o tempo todo para executar um processo que roda poucos minutos por dia. Isso gera custo fixo desnecessário e aumenta o esforço operacional com manutenção, segurança e monitoramento.

- **Azure Data Factory (ADF):** Apesar de ser uma solução robusta para pipelines complexos, é excessivo para o cenário atual. O custo e a complexidade não se justificam para extrações simples e diárias.

- **Container Docker em Azure Container Instances (ACI):** Oferece mais controle, mas adiciona complexidade com gerenciamento de imagens, registro de containers e orquestração. Para este caso, é mais do que o necessário.

- **Azure Logic Apps:** É uma boa ferramenta para automações, mas não atende bem à necessidade de lógica customizada em Python para extração incremental. Acabaria exigindo Azure Functions de qualquer forma.
