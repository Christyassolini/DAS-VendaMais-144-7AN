# ADR-002: Estratégia de Armazenamento Analítico

**Status:** Aceito | **Data:** 2026-03-13 | **Autores:** Equipe de Arquitetura — Turma DAS II / UNIVILLE

---

## Contexto

Após a extração diária dos dados do ERP (ver ADR-001), os dados brutos são armazenados no Azure Blob Storage (camada raw). Depois da transformação feita pela Azure Function, esses dados precisam ser persistidos em um repositório estruturado que funcione como fonte única de verdade para os dashboards do Power BI.

É necessário decidir qual serviço de banco de dados será utilizado como repositório analítico central, considerando:

- Modelo de dados: os dados do ERP são naturalmente relacionais (pedidos com itens, clientes, representantes, produtos, títulos financeiros). O modelo estrela (star schema) se encaixa bem no perfil analítico
- Padrão de consulta: análises por período, região, representante e cliente, com uso de agregações, filtros e JOINs entre tabelas fato e dimensão
- Volume estimado: ~3.500 pedidos/mês, 1.200 clientes e 18 representantes, com projeção de 2 a 5 GB nos primeiros 2 anos
- Integração: o Power BI Service precisa se conectar de forma nativa, com suporte a Import Mode e DirectQuery
- Rastreabilidade: cada carga deve permitir auditoria, indicando quando os dados foram inseridos, de qual extração vieram e se houve reprocessamento
- Custo: o projeto possui restrição orçamentária, então o custo precisa ser previsível e proporcional ao uso
- Segurança: dados financeiros e comerciais exigem controle de acesso, criptografia e logs de auditoria
- Equipe: já possui experiência com SQL relacional, então soluções muito diferentes disso devem ser avaliadas com cautela

---

## Decisão

Adotar o Azure SQL Database (tier Standard S2, 50 DTUs) como repositório analítico central da plataforma.

O banco será estruturado em modelo estrela (star schema), com tabelas fato por módulo (`fato_vendas`, `fato_estoque`, `fato_financeiro`, `fato_logistica`) e dimensões compartilhadas (`dim_cliente`, `dim_representante`, `dim_produto`, `dim_tempo`).

A carga será feita via UPSERT (MERGE em T-SQL) pela Azure Function de transformação, garantindo idempotência, ou seja, reprocessar o mesmo dia não gera duplicidade. Cada registro incluirá metadados de rastreabilidade (`data_carga`, `data_extracao`, `origem_arquivo`).

O Power BI irá se conectar usando Import Mode, com atualização diária agendada, garantindo melhor performance nos dashboards.

---

## Consequências

(+) **Modelo relacional familiar:** A equipe já domina T-SQL e modelagem estrela, o que reduz bastante a curva de aprendizado.

(+) **Integração direta com Power BI:** O conector entre Azure SQL e Power BI é estável, bem documentado e amplamente utilizado.

(+) **Rastreabilidade dos dados:** As colunas de metadados permitem identificar a origem de cada registro, facilitando auditoria, debug e reprocessamento.

(+) **Custo previsível:** O tier S2 atende bem o volume atual e permite escalar conforme necessário sem mudar a arquitetura.

(+) **Segurança já incluída:** O serviço oferece criptografia, controle de acesso, firewall e auditoria nativamente.

(-) **Não é um data warehouse dedicado:** Para volumes muito grandes ou análises extremamente complexas, outras soluções seriam mais adequadas. Para o cenário atual, isso não é um problema.

(-) **Custo fixo mensal:** Diferente de serviços serverless, o banco possui custo mesmo sem uso. Ainda assim, o valor é baixo e aceitável para o projeto.

---

## Alternativas Rejeitadas

- **Azure Cosmos DB (NoSQL):** Apesar de escalável e performático, não se encaixa bem com dados altamente relacionais. A modelagem ficaria mais complexa sem ganho real para o tipo de consulta necessário.

- **Azure Synapse Analytics:** É uma solução robusta para grandes volumes e cenários enterprise, mas possui custo e complexidade muito acima do necessário para este projeto.

- **Azure Database for PostgreSQL:** Tecnicamente viável, mas não traz vantagem clara em relação ao Azure SQL dentro de um ecossistema totalmente Microsoft. Também exigiria adaptação da equipe.

- **Apenas Azure Blob Storage (sem banco):** Reduz custo, mas compromete desempenho das consultas, integridade dos dados e rastreabilidade. Para dashboards com JOINs e filtros, um banco relacional é essencial.
